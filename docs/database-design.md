# Fuel Station Shift Reconciliation System
## Database Design

## 1. Purpose

This document defines the relational database design for the Fuel Station Shift Reconciliation System.

The database supports:

- Multiple fuel stations
- Multiple dispenser units
- DU serial number identification
- Configurable nozzles and fuel types
- Employees and effective-dated nozzle assignments
- Operational shifts
- Start and end totalizer receipts
- OCR results and manual corrections
- Fuel price history
- Nozzle-wise sales calculations
- Cash, UPI, card and credit collections
- Expenses and approved adjustments
- Employee-wise reconciliation
- Shift-level reconciliation
- Approval workflows
- Authentication and authorization
- Audit history

PostgreSQL is the primary production database.

Flyway will manage database schema migrations.

H2 may be used for automated tests where PostgreSQL-specific behaviour is not required.

---

## 2. Design Principles

The database design follows these principles:

1. Use normalized relational tables.
2. Retain historical configuration and transaction data.
3. Use effective dates for assignments and fuel prices.
4. Use database constraints to protect critical business rules.
5. Use `NUMERIC` instead of floating-point types for money and fuel quantities.
6. Use immutable transaction snapshots where master data can change later.
7. Retain original OCR values and manual corrections.
8. Record creation and modification metadata.
9. Use soft deactivation for master records where history must be preserved.
10. Avoid storing receipt image binary data directly in the database initially.
11. Use unique constraints to prevent duplicate business records.
12. Add indexes for common lookup and reporting operations.
13. Use explicit status columns for business workflows.
14. Use UTC timestamps for system events where possible.
15. Use database transactions for shift, payment and reconciliation operations.

## 3. Database Naming Conventions

### Tables

Use lowercase `snake_case`.

Examples:

```text
fuel_station
dispenser_unit
nozzle_assignment
receipt_nozzle_reading
employee_reconciliation
```

### Columns

Use lowercase `snake_case`.

Examples:

```text
station_code
du_serial_number
business_date
created_at
expected_sales_amount
```

### Primary keys

Use:

```text
id
```

Each table will generally have its own `BIGINT` generated identity primary key.

### Foreign keys

Use the referenced entity name followed by `_id`.

Examples:

```text
fuel_station_id
employee_id
shift_id
receipt_id
```

### Constraints

Suggested prefixes:

```text
pk_  primary key
fk_  foreign key
uk_  unique constraint
ck_  check constraint
```

Examples:

```text
pk_fuel_station
fk_nozzle_dispenser_unit
uk_dispenser_unit_du_serial_number
ck_payment_amount_non_negative
```

### Indexes

Use:

```text
idx_<table>_<columns>
```

Example:

```text
idx_shift_station_business_date
```

## 4. Common Audit Columns

Most business tables should contain:

```text
created_at
created_by
updated_at
updated_by
version
```

Recommended definitions:

```sql
created_at TIMESTAMP WITH TIME ZONE NOT NULL
created_by BIGINT
updated_at TIMESTAMP WITH TIME ZONE
updated_by BIGINT
version BIGINT NOT NULL DEFAULT 0
```

The `version` column supports optimistic locking.

Some append-only transaction or audit tables may not need `updated_at`, `updated_by`, or `version`.

---

## 5. Identifier Strategy

Use PostgreSQL identity columns:

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY
```

Example:

```sql
id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY
```

Reasons:

- Easy JDBC integration
- Supports large data volumes
- Database-managed identifier generation
- Compatible with PostgreSQL
- More explicit than the older `SERIAL` type

External APIs should not expose sequential database IDs where this creates security or integration concerns.

A future public identifier may be added:

```text
public_id UUID
```

For the first implementation, numeric internal identifiers are sufficient.

## 6. Numeric Precision

### Monetary values

Use:

```sql
NUMERIC(19, 2)
```

Examples:

```text
ATOT
Payment amount
Expense amount
Expected sales amount
Difference amount
```

### Fuel volume

Use:

```sql
NUMERIC(19, 3)
```

Examples:

```text
VTOT
Quantity sold
Petrol volume
Diesel volume
```

### Fuel price

Use:

```sql
NUMERIC(12, 3)
```

This supports prices with three decimal places if required.

### OCR confidence

Use:

```sql
NUMERIC(5, 2)
```

Valid range:

```text
0.00 to 100.00
```

### Java mapping

All these numeric values must map to:

```java
java.math.BigDecimal
```

The application must not use `double` or `float` for financial or fuel calculations.

---

## 7. Date and Time Strategy

Use:

```sql
DATE
```

for business dates.

Use:

```sql
TIMESTAMP WITH TIME ZONE
```

for actual system timestamps.

Examples:

```text
business_date        DATE
started_at           TIMESTAMP WITH TIME ZONE
uploaded_at          TIMESTAMP WITH TIME ZONE
calculated_at        TIMESTAMP WITH TIME ZONE
```

The backend should store timestamps in UTC and convert them for display based on the configured station or user time zone.

## 8.7 `nozzle_assignment`

Stores effective-dated nozzle-to-employee assignments.

Columns:

| Column | Type | Required | Description |
|---|---|---:|---|
| id | BIGINT | Yes | Primary key |
| nozzle_id | BIGINT | Yes | Assigned nozzle |
| employee_id | BIGINT | Yes | Assigned employee |
| effective_from | DATE | Yes | Assignment start date |
| effective_to | DATE | No | Assignment end date |
| active | BOOLEAN | Yes | Active flag |
| created_at | TIMESTAMPTZ | Yes | Creation timestamp |
| created_by | BIGINT | No | Creating user |
| updated_at | TIMESTAMPTZ | No | Update timestamp |
| updated_by | BIGINT | No | Updating user |
| version | BIGINT | Yes | Optimistic-lock version |

Check constraint:

```text
effective_to IS NULL OR effective_to >= effective_from
```

Business rule:

A nozzle must not have overlapping assignments.

Because standard unique constraints cannot fully prevent overlapping date ranges, this must be enforced through:

- Application validation
- Transaction locking
- Optionally, a PostgreSQL exclusion constraint in a later migration

# 19. Delete and Update Behaviour

Master records with transaction history should generally not be physically deleted.

Use:

```text
active = false
```

for:

```text
fuel_station
dispenser_unit
nozzle
employee
fuel_type
fuel_price
application_user
role
cash_denomination
```

Recommended foreign-key behaviour:

### `ON DELETE RESTRICT`

Use for important master-to-transaction relationships.

Examples:

```text
fuel_station → shift
employee → shift_employee
nozzle → fuel_sale
```

### `ON DELETE CASCADE`

Use only for tightly owned child records that cannot exist independently.

Examples may include:

```text
receipt → ocr_field_result through OCR attempt
reconciliation → employee_reconciliation
```

However, even cascades should be used cautiously because transaction and audit data should rarely be physically deleted.

Normal application operations should not delete completed shifts, receipts, reconciliations or audit records.

# 21. Unique Business Constraints

The database should enforce these rules wherever possible:

1. Organization code is unique.
2. Station code is unique within an organization.
3. Normalized DU serial number is unique.
4. Nozzle number is unique within a dispenser unit.
5. Employee code is unique within an organization.
6. One nozzle assignment cannot overlap another assignment.
7. One active fuel price period cannot overlap another for the same station and fuel type.
8. Shift number is unique for DU and business date.
9. A shift has at most one start receipt.
10. A shift has at most one end receipt.
11. A receipt has at most one reading per nozzle.
12. A shift has one fuel-sale record per nozzle.
13. A shift and employee have at most one denomination row per denomination.
14. A reconciliation has one employee reconciliation per employee.
15. Usernames are unique within an organization.

Rules involving date-range overlap may require PostgreSQL exclusion constraints or application-level transactional validation.

# 23. Concurrency and Optimistic Locking

Important mutable tables should contain:

```text
version BIGINT NOT NULL DEFAULT 0
```

The application should use optimistic locking when updating:

```text
fuel_station
dispenser_unit
nozzle
employee
nozzle_assignment
fuel_price
shift
receipt
payment_entry
cash_denomination_entry
credit_sale
adjustment
application_user
```

Example update pattern:

```sql
UPDATE shift
SET status = ?,
    updated_at = ?,
    updated_by = ?,
    version = version + 1
WHERE id = ?
  AND version = ?;
```

If no row is updated, another user changed the record and the operation must be retried or rejected.

- Be immutable after deployment
- Use deterministic SQL
- Avoid environment-specific data
- Fail safely
- Be tested on an empty database
- Be tested when upgrading from the previous version

Never edit an already deployed migration.

Create a new migration to change an existing schema.

# 31. Security Considerations

The database user used by the application should:

- Not be a PostgreSQL superuser
- Have access only to the application schema
- Use a strong password or managed secret
- Connect using TLS in production
- Not share credentials with administrators
- Have only the permissions required by the application

Flyway may use:

- The same database user initially, or
- A separate schema-owner account in mature deployments

Sensitive values such as database passwords must not be committed to Git.
